Team Dashboard
 
Many customers experience a typical progression: light usage early in the NimbleDroid trial gives way to lots of Scenarios being profiled for multiple uploads a day. Sometimes it can be helpful to to zoom out and review performance trending over a longer period of time. It can also be useful to bubble up any new regressions that have shown up in recent builds.
 
To that end we have created the Team Dashboard page. The dashboard aggregates trending for each Scenario for speed, frame drop rate, and memory, network, and disk usage. There is a visual indicator of large changes from upload to upload, as well as inicators of when Smart Alerts are triggered. At the top we report aggregate data about uploads, and show any new regression detected with the latest upload.
 
 
 
The dashboard can be accessed with a button at the top of the App Info page.
 
Smart Alerts
 
A key ability of NimbleDroid is notifying customers when a regression is detected for a new upload. In the past NimbleDroid has provided alerts for slowdowns based on user-entered parameters. This has been effective but ultimately time-consuming to manage; each Scenario must be configured manually, and gradual changes to Scenario nominal times have required alerts to be manually updated.
 
NimbleDroid now offers Smart Alerts, a self-learning algorithm that analyzes your Scenarios to determine what is normal timing and automatically generates an alert for slowdowns. We will soon add the ability to tweak the sensitivity of Smart Alert thresholds for even greater control.
 
 
A slow upload that has triggered a smart alert (shown in red)
 
Smart Alerts are enabled by default for all Scenarios when you upload a new app via the site or API, and you can choose Slack or email for notifications. Smart Alerts can be disabled or enabled on the App Settings page, and for customers that prefer we still offer the legacy manual alert feature which can be used with or without Smart Alerts.
 
 
Frame Drop Rate
 
We now measure performance of drawing frames on the screen for Scenarios in Android. The term we use is Frame Drop Rate, which is inspired by Facebook's talk on drawing performance (You can check that out here. To sum up, we report the percentage of Scenario time that is spent dropping 2 consecutive frames or more. So for a 2 second Scenario with a 10% FDR, a total of 200ms is spent in a laggy state for the user. You can find this information on the App Info, Scenario Detail, or Team Dashboard pages.
 
 
Frame Drop Rate is not measured for Cold Startup due to the inherent lack of interactivity.
